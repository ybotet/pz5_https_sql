## Описание проекта

Этот проект реализует защищенное HTTPS-соединение и защиту от SQL-инъекций для микросервисов `auth` и `tasks`.

**Цели практической работы:**

- Настроить TLS-терминацию с использованием Nginx
- Продемонстрировать уязвимость SQL injection
- Исправить уязвимость с помощью параметризованных запросов
- Реализовать безопасное хранение данных в PostgreSQL

## Часть A: HTTPS/TLS

### Генерация сертификатов

Для разработки использованы самоподписанные сертификаты:

```bash
mkdir -p deploy/tls
cd deploy/tls

openssl req -x509 -newkey rsa:2048 -nodes \
  -keyout key.pem \
  -out cert.pem \
  -days 365 \
  -subj "//CN=localhost"

```

### Конфигурация Nginx

Файл deploy/tls/nginx.conf:

```bash
events {}

http {
    server {
        listen 8443 ssl;
        server_name localhost;

        ssl_certificate /etc/nginx/tls/cert.pem;
        ssl_certificate_key /etc/nginx/tls/key.pem;

        location / {
            proxy_pass http://tasks:8082;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto https;
            proxy_set_header X-Request-ID $http_x_request_id;
            proxy_set_header Authorization $http_authorization;
        }
    }
}
```

## Демонстрация уязвимости

Уязвимый код (конкатенация строк):

```go

// НЕПРАВИЛЬНО: уязвимо для SQL injection
func (r *PostgresTaskRepository) SearchByTitleVulnerable(title string) ([]models.Task, error) {
    query := fmt.Sprintf("SELECT * FROM tasks WHERE title = '%s'", title)
    rows, err := r.db.Query(query)
    // ...
}
```

Эксплуатация уязвимости:

```bash

# URL: /v1/tasks/search/vulnerable?title=' OR '1'='1
curl -k "https://localhost:8443/v1/tasks/search/vulnerable?title=%27%20OR%20%271%27%3D%271" \
  -H "Authorization: Bearer demo-token"
```

Результат: Возвращаются ВСЕ задачи, так как запрос превращается в:

```sql

SELECT * FROM tasks WHERE title = '' OR '1'='1'
-- Условие '1'='1' всегда истинно → все записи
```

### Исправление параметризацией

Безопасный код (параметризованный запрос):

```go

// ПРАВИЛЬНО: защита от SQL injection
func (r *PostgresTaskRepository) SearchByTitle(title string) ([]models.Task, error) {
    query := "SELECT * FROM tasks WHERE title = $1"
    rows, err := r.db.Query(query, title)  // title передается как параметр
    // ...
}
```

## ** Контрольные вопросы**

### **1. Какие свойства даёт TLS соединению?**

TLS (Transport Layer Security) обеспечивает три фундаментальных свойства безопасности. **Конфиденциальность** гарантирует, что данные шифруются и не могут быть прочитаны третьими лицами при передаче. **Целостность** позволяет обнаружить любое изменение данных в пути — если злоумышленник попытается модифицировать трафик, это будет немедленно выявлено. **Аутентификация** дает клиенту возможность убедиться, что он соединяется именно с подлинным сервером, а не с поддельным, благодаря проверке цифрового сертификата.

### **2. Почему самоподписанный сертификат не подходит для реального продакшна?**

Самоподписанные сертификаты не подходят для production по нескольким причинам. Во-первых, браузеры и клиенты не доверяют им по умолчанию и выводят предупреждения об угрозе безопасности, что создает плохой пользовательский опыт. Во-вторых, отсутствует проверка подлинности центра сертификации (CA) — любой может создать самоподписанный сертификат для любого домена. В-третьих, такие сертификаты уязвимы для атак Man-in-the-Middle, так как злоумышленник может подменить сертификат без обнаружения.

### **3. В чём отличие TLS-терминации на NGINX от TLS в приложении?**

Основное различие заключается в месте обработки шифрования и управлении сертификатами. При TLS-терминации на **NGINX** все HTTPS-соединения обрабатываются централизованно на прокси-сервере, что обеспечивает лучшее использование ресурсов, упрощает управление сертификатами (обновление без перезапуска приложений) и снижает нагрузку на бэкенд-сервисы. При реализации **TLS непосредственно в приложении** каждый микросервис самостоятельно обрабатывает шифрование, что дает больше контроля, но увеличивает сложность, требует хранения ключей в каждом сервисе и снижает производительность из-за дублирования операций шифрования.

### **4. Как возникает SQL-инъекция?**

SQL-инъекция возникает, когда пользовательский ввод напрямую конкатенируется в строку SQL-запроса без должной обработки. Например, код вида `"SELECT * FROM users WHERE name = '" + userName + "'"` позволяет злоумышленнику ввести специально оформленную строку, например `' OR '1'='1`, которая изменяет структуру исходного запроса. В результате запрос превращается в `SELECT * FROM users WHERE name = '' OR '1'='1'`, что возвращает все записи из таблицы, так как условие `'1'='1'` всегда истинно. Это может привести к несанкционированному доступу к данным, их модификации или даже удалению.

### **5. Почему параметризованный запрос защищает от SQLi?**

Параметризованные запросы защищают от SQL-инъекций благодаря разделению кода SQL и пользовательских данных. Сначала база данных получает и компилирует шаблон запроса с плейсхолдерами (например, `$1`, `$2`), определяя структуру команды. Затем уже подготовленный запрос выполняется с подстановкой переданных параметров **исключительно как данных**, которые не интерпретируются как исполняемый код. Даже если параметр содержит SQL-синтаксис, он не будет выполнен — база данных воспримет его как обычное строковое значение. Например, при запросе с параметром `"'; DROP TABLE users; --"` этот текст будет рассматриваться как буквальный поиск, а не как команда удаления таблицы.

### **6. Почему детали ошибок БД нельзя показывать клиенту?**

Детали ошибок базы данных нельзя показывать клиенту по соображениям безопасности. Такие сообщения часто содержат техническую информацию: имена таблиц и колонок, структуру запросов, версию СУБД и фрагменты данных. Эта информация значительно облегчает злоумышленнику подготовку целенаправленных атак. Кроме того, многие стандарты безопасности (PCI DSS, GDPR) прямо требуют скрывать технические детали от конечных пользователей. Правильный подход — логировать полную информацию об ошибке на стороне сервера для отладки, а клиенту возвращать общее сообщение, например "Внутренняя ошибка сервера", не раскрывая причин сбоя.
